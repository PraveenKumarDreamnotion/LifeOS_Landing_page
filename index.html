What is LifeOS

A personal AI companion that lives on your own machine.

You talk to Yogi like a person. It sets reminders, answers questions with live sources, and keeps an eye on your inbox - and it never sends your life to someone else's server unless you explicitly turn a feature on, with your own key.

Voice-first, not voice-only

Speak or type. Yogi listens, shows a live transcript you can edit, and replies out loud or on screen - whichever suits the moment.

Private by architecture

Reminders, chats, and settings live in a local SQLite file. A built-in network filter blocks every outbound request until you enable a cloud feature.

Fast desktop experience

A native Windows app with a system-tray presence and a global hotkey, so Yogi is one keystroke away from anywhere.

The problem & the idea

The most helpful assistants ask for the most trust.

A voice assistant that sets reminders, answers questions, and reads your inbox has to touch the most personal parts of your day. Almost every one solves that by shipping it all to a server. LifeOS takes the opposite bet.

The productivity tax

Reminders live in one app, questions in another, email in a third. Switching between them all day is its own kind of work - and the tools that could unify it want an account first.

Why the usual assistants fall short

Cloud assistants keep an always-open connection, often an always-listening mic, and ask you to trust a policy document. They can't run offline, and they confidently make up live facts.

Why local-first is the fix

Your reminders, chats, email, and voice are your life. A promise can be broken; code can't. LifeOS runs on your machine and proves it - the network is blocked by default, in code.

The idea

Keep the intelligence, move the trust boundary to your device.

Everything runs on-device by default. Cloud intelligence is strictly opt-in, one toggle per capability, and always uses your own OpenAI key - billed to you, gated in the main process so the UI can't fake consent. You get a real companion without a data trail.

Why LifeOS exists

The workflow is fragmented. LifeOS puts it back together.

Every tool nails one slice

A launcher, a dictation box, an inbox agent - each is excellent at its one job, and none of them talk to each other.

So you keep switching apps

Reminders in one window, voice in another, email in a third. The context-switching is its own tax on your day.

And most of them send it to a server

The richer the assistant, the more of your words leave your machine by default. A privacy policy is a promise, not a guarantee.

LifeOS is the one that combines them - locally

Voice, reminders, chat, web answers, and email fold into a single Windows app that works offline and blocks the network by default, in code. Cloud is opt-in, with your own key.

Listen & understand

Capture → 16 kHz PCM

An AudioWorklet down-samples the mic to 16 kHz and streams frames to the main process. Audio is never written to disk.

Speech-to-text

Transcribed on-device by the bundled sherpa-onnx Zipformer - or, if you opt in, by OpenAI gpt-4o-transcribe with automatic fallback to sherpa.

Intent routing

A local router answers time / greeting / settings instantly with no model. Otherwise the gated LLM (gpt-4o-mini, strict JSON) classifies the turn - and if there's no key, an offline parser takes over.

Reminder

Re-routed to the local parser → a confirmation card. The LLM never creates it; nothing saves until you confirm by click or “yes.”

Chat / question / research

Answered by the model; live-fact turns trigger a web search (gpt-4o-mini-search-preview) and return an answer with sources.

Email tools

When Gmail is connected, a new email is summarized into its own chat with action items - grounded, local, opt-in.

How it works

Frontend and backend, in plain terms.

The renderer is a thin, sandboxed React shell. The Node main process holds the reliability spine - the scheduler, the parser, the database, and the single gate that lets anything reach the cloud.

Frontend

Framework
React 19 + TypeScript, sandboxed (no Node access), bundled by electron-vite.
Desktop UI
Four windows - main, reminder popup, voice launcher, and a hidden audio host - each its own HTML entry, all sharing core/.
State
Plain React hooks: useSettings, useReminders, useConversation, useSessions, useSpeech. No Redux, no external store.
Routing
State-based navigation - a left rail switches chat / schedules / history / settings. No router library for four screens.
Components
Chat transcript, proposal & dispatch cards, a dependency-free XSS-safe Markdown renderer, a real focus-trap Modal, and a live mic waveform.
Styling
Plain CSS custom properties - no Tailwind, no CSS-in-JS. Light/dark via prefers-color-scheme and a data-theme override.
Experience
An offline/online chip, a paused banner, live-ticking relative times, an aria-live transcript, and mid-speech mic interruption.

Backend

Business logic
A ConversationEngine runs each turn: local router → gated LLM → validate → branch → persist → broadcast, guaranteeing exactly one chat:done per turn.
Reminder engine
A pure, total parseReminder (56 test fixtures) → an action dispatcher (propose → confirm, 90s timeout = cancel) → a single verified writer. The LLM never actuates.
Scheduling
Wall-clock authoritative: a 30-second reconcile query - never a far-future timer - plus powerMonitor resume and a missed-while-closed catch-up.
Voice pipeline
Native offline STT (sherpa-onnx), and a speak coordinator that streams OpenAI audio over MSE with a blob and Windows-voice fallback.
OpenAI calls
A provider registry gates each seam (LLM / STT / TTS / search) on enable + key + consent, re-run per turn. The key is read at call time and never crosses IPC.
Database
SQLite via node:sqlite (WAL), 18 tables at schema v8, reached only through parameterized repositories.
Background services
The scheduler tick and Gmail sync scheduler run in the never-throttled main loop and shut down cleanly, checkpointing the WAL on quit.

Features I'm most proud of

Floating desktop launcher

A frameless, always-on-top widget summoned by a global hotkey. It shows over any app, starts listening immediately, and never steals focus - and it shares one live conversation with the main window and the reminder popup.

Natural-language reminders

Say “remind me in ten minutes to drink water.” A deterministic local parser (56 test fixtures) reads it, proposes a card, and waits - the model never creates the reminder, and nothing is saved until you confirm by click or by saying “yes.”

Local-first, enforced in code

A default-deny network filter blocks every outbound request until you switch on a cloud feature. Your key is DPAPI-encrypted and never crosses the app's internal boundary. Privacy you can verify, not just a policy you're asked to trust.

Edge cases handled

Interruption A reminder fires mid-conversation

The launcher stops speaking, hides, and snapshots your session; the reminder popup takes over. When you're done, it re-opens on the same chat and re-reads the interrupted reply from the start.

Offline No internet at all

Reminders, time, greetings, help, and “show schedules” all work with zero network. On-device speech-to-text and Windows voices keep running; only genuine reasoning is unavailable.

No API key Cloud features off

The chip reads “Offline · on-device.” Reminder-shaped input still routes to the local parser; a request that truly needs the cloud gets an honest “connect OpenAI” notice - never a fake answer.

Network failure The cloud call dies

Cloud speech-to-text falls back to sherpa; cloud voice falls back to Windows; a failed chat or search returns an honest message - never a hang, never a fabricated success.

Mis-hear Speech recognition slips

Offline STT can hear “remained me” for “remind me.” A bounded edit-distance normalizer canonicalizes the mangled cue so the reminder still parses. If the model is missing, typing still works.

Email off Gmail sync disabled

Gmail is opt-in and off by default. With it disabled there are no email chats, no sync, and no Google requests - the rest of the assistant is completely unaffected.

Your data, on your disk

Where everything lives after you install.

LifeOS keeps everything in one local SQLite file you can open and read yourself. No cloud bucket, no hidden config sprawl - just a database on your machine.

Default location %APPDATA%\LifeOS\lifeos.db (development builds use %APPDATA%\lifeos\) 
%APPDATA%\LifeOS\
├─ lifeos.db        main SQLite database - all your data
├─ lifeos.db-wal    write-ahead log  ·  do not edit
├─ lifeos.db-shm    shared-memory index  ·  do not edit

SQLite via node:sqlite, WAL mode, foreign keys on - schema user_version 8, 18 tables. Tip: Settings → Privacy → Open data folder jumps straight here.

Inspect the database

Open lifeos.db with any SQLite viewer - DB Browser for SQLite is the easy pick. Quit LifeOS first (or open read-only) so it isn't writing the WAL while you browse.

Don't edit these by hand

Leave the -wal / -shm sidecars alone, and never edit the *_ciphertext columns - your OpenAI key and Gmail tokens are DPAPI-encrypted there, so a manual edit just corrupts them.

Reset safely

Settings → Danger zone → type RESET deletes lifeos.db and its sidecars, then relaunches. Uninstalling the app deliberately keeps this folder, so an accidental uninstall never loses your data.

On credentials: secrets are encrypted with Electron safeStorage, which uses the Windows Data Protection API (DPAPI). The ciphertext lives in the settings table - LifeOS does not use the Windows Credential Manager. Decrypted values exist only in the main process and never cross into the UI.
What's next

An honest roadmap - and what still needs work.

Everything below is on the roadmap, not in today's build. Marked plainly, because a showcase that overstates itself isn't much of a showcase.

Future features

  • Long-term memory - durable facts about you; the schema is wired, the extraction & recall aren't.
  • Local LLM via Ollama - the same provider seam, with zero network and zero cost.
  • Streaming replies - the channel is reserved; today replies arrive whole.
  • Recurring by voice, beyond weekly - the editor already does monthly/yearly; the parser is next.
  • Auto-update - latest.yml already ships; wiring electron-updater is the step left.
  • Import / export your data as JSON, and more tools (weather, opt-in calendar).
  • macOS & Linux - core/ is already pure TypeScript; only the Electron layer changes.

Biggest improvements needed

  • Ship-ready packaging - code-sign the build and QA it on a clean Windows VM to end the SmartScreen warning.
  • Long-term memory is the real gap between “chatbot” and “companion.”
  • Move STT decode off the main thread into a worker for isolation.
  • Repay UI debt - a shared button primitive, retire the legacy chat screen, keyboard nav on the popup menu.
  • Log retention & an in-app log viewer - neither is wired yet.
  • Streaming + inline cloud-error recovery for a smoother feel under flaky networks.

If this became a startup

  • Audience - privacy-conscious Windows power users first; later, teams in regulated fields that can't ship data to a vendor cloud.
  • Model - the app stays free & open-source, bring-your-own-key. Monetize an optional managed layer, never the core.
  • Pricing ideas - a low monthly “managed cloud” tier (a hosted relay so non-technical users skip key setup), and a team tier with policy controls & local-model deployment.
  • Enterprise - “provably on-device” is a compliance story, not just a feature.
  • Scaling - because the cloud is opt-in, growth is mostly client-side; no per-user server cost until someone chooses the managed tier.

Framed as hypotheticals - LifeOS is a free, open-source project today.

FAQ

Honest answers.

Do my reminders fire when LifeOS is closed?
Reminders fire while LifeOS is running, including when it’s tucked away in the system tray (closing the window with ✕ keeps it running by default). If you fully quit LifeOS or shut down your PC, reminders won’t fire until you open it again - and when you do, it tells you what you missed rather than firing old alarms late. This is deliberate: LifeOS refuses to install a hidden background service.
Is my data sent anywhere?
Not by default - LifeOS makes zero network requests with its default settings, enforced in code by a filter that blocks anything not on an allowlist. The only exception is when you enable a cloud AI feature and add your own OpenAI key; then the text of that command (plus the current date and time) goes directly to OpenAI under your account. Your reminders, history, settings, and audio are never sent.
Do I need an OpenAI key?
No. LifeOS works fully offline for reminders, time, schedules, on-device speech-to-text, and offline voices. A key is optional and unlocks smarter chat, cloud speech-to-text, more natural voices, web search, and email summaries. You bring your own key and OpenAI bills you directly - the app itself is free.
Does Yogi listen in the background or use a wake word?
No. There’s no wake word and no always-on listening. Yogi only records when you press the mic or the global hotkey, and audio is transcribed on your device and never written to disk. A wake word is on the roadmap but is not in this release.
Which platforms are supported?
Windows 10 (version 1809 or newer) and Windows 11, 64-bit only. macOS, Linux, mobile, and web are not available in this release. The core is written in portable TypeScript, so other platforms are a future possibility.
Why does Windows warn me during install?
The installer isn’t code-signed yet, so Microsoft SmartScreen shows a generic “Windows protected your PC” warning. Click More info → Run anyway to proceed. Code signing is planned for a future release.
How do I remove my data?
Open Settings → Danger Zone → “Reset local data.” It permanently deletes the reminders, history, and settings in your LifeOS data folder and nothing else on your device. Note that uninstalling the app does not delete this folder - that’s intentional, so an accidental uninstall never loses your data.
